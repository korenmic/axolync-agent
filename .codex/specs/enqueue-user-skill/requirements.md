# Requirements

## Introduction

The enqueue user skill adds the currently intended undone tasks to the local execution queue, then reports updated queue health. It standardizes queue population without running implementation work.

The source seed is `project-seeds/03-enqueue-user-skill.md`.

## Requirements

### Requirement 1: User Skill Placement And Safety

**User Story:** As an agent maintainer, I want `$enqueue` to live in the agents repo user-skill source area, so it can be installed consistently and used across workspaces.

#### Acceptance Criteria

1. WHEN the skill is created THEN the skill source SHALL live under `axolync-agent/skills-user/enqueue`.
2. WHEN the skill is invoked THEN `$enqueue` SHALL be the canonical invocation.
3. WHEN the skill runs THEN it SHALL mutate only the local queue file needed for enqueueing.
4. WHEN the skill runs THEN it SHALL NOT implement tasks, run TACTIC, mark tasks done, or reorder existing queue records.
5. WHEN the skill completes THEN it SHALL run queue-status reporting and state how many undone tasks were added in that enqueue session.

### Requirement 2: Context-Scoped Task Selection

**User Story:** As a user enqueueing work from a live discussion, I want enqueue to add only the tasks that are relevant to the current conversation, so unrelated unchecked backlog items are not pulled into the queue accidentally.

#### Acceptance Criteria

1. WHEN the user names specific tasks THEN the skill SHALL enqueue only those requested undone tasks.
2. WHEN the user says "the tasks", "this seed", "that spec", or similar context-dependent wording THEN the skill SHALL resolve the intended tasks from the current conversation context.
3. WHEN a source file contains unrelated unchecked tasks outside the requested scope THEN the skill SHALL NOT enqueue those unrelated tasks.
4. WHEN the context is insufficient to identify the intended task set safely THEN the skill SHALL stop and ask for a concise clarification.
5. WHEN tasks were just generated by seed-to-spec in the current session THEN those generated tasks MAY be treated as the intended enqueue scope.

### Requirement 3: Source Types

**User Story:** As an agent enqueueing different work sources, I want the skill to handle spec tasks, backlog tasks, and context-only tasks with the correct queue record type.

#### Acceptance Criteria

1. WHEN enqueueing a task from a spec `tasks.md` file THEN the skill SHALL create a by-reference queue record.
2. WHEN enqueueing a task from a backlog `tasks.md` file THEN the skill SHALL create a by-reference queue record.
3. WHEN enqueueing tasks from multiple specs in one request THEN the skill SHALL append records for the requested undone tasks from each spec.
4. WHEN an approved task exists only in conversation context and has no authoritative source file THEN the skill SHALL create a by-value queue record.
5. WHEN creating a by-value record THEN the skill SHALL include enough task text for a later tactic runner to execute it without relying on the original conversation.

### Requirement 4: Undone-Only Default

**User Story:** As a user, I want enqueue to add unfinished work by default, so completed tasks are not reintroduced as runnable queue items.

#### Acceptance Criteria

1. WHEN source tasks are unchecked THEN the skill SHALL treat them as eligible undone tasks.
2. WHEN source tasks are checked or otherwise completed THEN the skill SHALL skip them by default.
3. WHEN the user explicitly requests a completed task as an audit record THEN the skill MAY enqueue it with clear task text.
4. WHEN a partially completed spec is supplied THEN only requested unchecked tasks SHALL be enqueued.

### Requirement 5: Duplicate Prevention

**User Story:** As an agent maintaining the queue, I want enqueue to avoid adding the same source task twice, so the queue stays runnable without duplicate work.

#### Acceptance Criteria

1. WHEN a requested source task is already present in the queue THEN the skill SHALL skip appending a duplicate record.
2. WHEN checking duplicates for by-reference tasks THEN the baseline identity SHALL be source path plus task number/title.
3. WHEN a stronger source task-id exists in the future THEN the skill SHALL prefer it over task text matching.
4. WHEN comparing source task identity THEN the skill MAY normalize Markdown wrappers, checklist markers, backticks, and whitespace.
5. WHEN comparing source task identity THEN the skill SHALL NOT use broad fuzzy matching that treats merely similar wording as the same task.
6. WHEN duplicates are skipped THEN the final response SHALL report the skipped count.

### Requirement 6: Queue File Creation And Append Format

**User Story:** As a user in a workspace without an initiated queue, I want enqueue to create the standard queue file, so queue setup does not require a separate manual step.

#### Acceptance Criteria

1. WHEN `<workspace-root>/.codex/local-task-queue.md` does not exist THEN the skill SHALL create it with the standard local queue header and `## Queued Items` section.
2. WHEN the queue exists THEN the skill SHALL preserve all existing records and append new records only.
3. WHEN appending records THEN the skill SHALL generate the next qid after the current highest queue id.
4. WHEN appending Markdown records THEN the skill SHALL use the existing format with heading, status, source, and task fields.
5. WHEN appending multiple records THEN the skill SHALL keep deterministic append order from the resolved task source order.

### Requirement 7: Queue-Status Integration

**User Story:** As a user, I want enqueue to immediately show updated queue health, so I can verify the queue state after new records are added.

#### Acceptance Criteria

1. WHEN enqueue completes THEN it SHALL run `$queue-status verbose` or the equivalent queue-status parser mode.
2. WHEN verbose queue-status support is unavailable THEN the skill SHALL report that limitation clearly.
3. WHEN queue-status output is shown THEN enqueue SHALL add a manual note stating how many undone tasks were added in the current session.
4. WHEN no tasks were appended THEN the manual note SHALL state that zero undone tasks were added.
5. WHEN duplicates were skipped THEN the manual note SHALL include the duplicate/skipped count.

### Requirement 8: Regression Coverage

**User Story:** As a maintainer, I want deterministic tests for enqueue behavior, so queue population does not regress as queue formats evolve.

#### Acceptance Criteria

1. WHEN tests are added THEN they SHALL cover a new spec with multiple unchecked tasks.
2. WHEN tests are added THEN they SHALL cover an existing partially done spec with a specific newly requested task.
3. WHEN tests are added THEN they SHALL cover multiple specs in one request.
4. WHEN tests are added THEN they SHALL cover backlog tasks.
5. WHEN tests are added THEN they SHALL cover duplicate prevention and qid continuation.
6. WHEN tests are added THEN they SHALL cover missing queue auto-creation.
7. WHEN tests are added THEN they SHALL cover context-only by-value tasks without requiring the literal phrase `by-value`.
