# Design

## Overview

`$enqueue` is a user skill that appends currently intended undone tasks to the local execution queue, then invokes queue-status reporting. It is a queue-population workflow, not an implementation workflow.

The design keeps three responsibilities separate:

- Skill wrapper: resolves user intent and enforces safety rules.
- Enqueue script/helper: parses task sources, prevents duplicates, creates/appends queue records.
- Queue-status integration: reports final queue health and leaves the source queue as the machine-readable authority.

## Skill Location

The skill source lives at:

`axolync-agent/skills-user/enqueue`

The canonical invocation is `$enqueue`.

## Queue Target

The default queue target is:

`<workspace-root>/.codex/local-task-queue.md`

If the queue file is absent, enqueue creates it with the established local header and an empty `## Queued Items` section. It then appends records to that section.

## Intent Resolution

The skill scopes work from current conversation context. It must not scan every visible spec/backlog and enqueue every unchecked item.

Supported intent sources:

- a specific `tasks.md` path
- a group of spec `tasks.md` paths
- specific task numbers or titles in an existing spec
- backlog `tasks.md` entries
- tasks just generated by the current `$s2s` workflow
- context-only approved task text with no source file

If a phrase like "the tasks" is used, the skill resolves it from the current discussion. If the context does not identify the scope clearly, it asks for clarification.

## Source Parsing

For by-reference source files, parse Markdown checklist tasks. The parser should capture:

- source path
- task number when present
- task title/text
- checked or unchecked state
- stable task identity derived from source path plus task number/title

Unchecked tasks are eligible by default. Checked tasks are skipped unless the user explicitly asks to enqueue them as audit records.

For context-only by-value tasks, the skill stores the task text directly in the queue record with an inline source label. The task text must be sufficient for later execution.

## Duplicate Detection

Queue duplicate detection prevents re-adding the same source task.

Baseline duplicate key for by-reference records:

```text
canonical source path + normalized task number/title
```

Normalization may remove obvious Markdown wrappers, checklist markers, backticks, and whitespace differences. It must not use broad fuzzy matching or treat merely similar wording as identical.

If source task files later expose a stronger task-id, the duplicate key should use that id. This source task identity is separate from `qid`, which is only the queue record identity.

By-value duplicates are detected conservatively from normalized task text and source label.

## Append Model

The queue record format is:

```markdown
### Q-483
- Status: `queued`
- Source: [tasks.md](/c:/path/to/tasks.md)
- Task: `1. Do the thing`
```

By-value records use a source label such as:

```markdown
- Source: `inline procedural queue task`
```

New qids continue from the highest existing qid in the active queue file. Append order follows resolved task order.

## Queue-Status Integration

After appending, enqueue invokes `$queue-status verbose` or the equivalent parser mode once implemented. The queue-status output remains the global queue-health authority.

Enqueue then adds a separate manual session note:

```text
Added N undone tasks in this enqueue session.
Skipped D duplicate tasks.
```

This note is intentionally local to the enqueue session and is not persisted as a machine-readable artifact.

## No-Op Sessions

If all requested tasks are done, duplicate, or out of scope, enqueue does not edit the queue. It reports why and may still run queue-status for visibility. The manual note should say zero tasks were added.

## Testing

Tests should use temporary queue fixtures and source task files. They should cover:

- new spec with multiple unchecked tasks
- partially done spec with a specific requested task
- multiple specs in one request
- backlog tasks
- duplicate prevention
- qid continuation
- missing queue auto-creation
- context-only by-value tasks
- checked tasks skipped by default

## Self-Review Notes

- The design prevents broad discovery of unrelated undone tasks.
- The design keeps enqueue as the only mutating step and queue-status as reporting-only.
- The design distinguishes source task identity from queue record qid.
- The design keeps by-value enqueueing possible without requiring literal `by-value` wording.
- The design supports queue auto-creation as requested.
